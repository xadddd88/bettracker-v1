# BetTracker AI — Engineering Manifesto v1.0
> Core principles that govern every engineering decision.

---

## From the Founder

We are not building another betting site.  
Our goal — create the best AI platform for decision-making in betting.

The core value is not predictions.  
The core value — help the user make better decisions.

## Mission

BetTracker AI must become the tool a user opens **before** placing money.  
Not after. Not for statistics. But before making a decision.

## Development Principles

**1. We build a product.** Not a site. Not an MVP. Not a set of screens. Every line of code must move us toward a long-term platform.

**2. Architecture over speed.** Better to spend one extra day now than rewrite for three weeks in six months.

**3. Every feature must answer:** What user problem does it solve? If no answer — the feature is not needed.

**4. Beautiful code is not enough.** Code must be: readable, scalable, testable, documented.

**5. The product must be useful without AI.** AI amplifies the product. It must not be the only reason for its existence.

## The Primary Object of the System

Not the bet.  
Not the user.  
Not the bankroll.

**The primary object is the Decision.**

A match is analyzed.  
The user makes a decision.  
The bet becomes a consequence of the decision made.

All architecture must account for this.

## What We Never Do

- Write temporary code "we'll rewrite it later"
- Create technical debt for the sake of speed
- Copy competitors without understanding why
- Complicate UX
- Write magic values
- Mix business logic with UI

## Roles

| Role | Responsible | Scope |
|------|-------------|-------|
| **Founder** | Дима | Final decisions |
| **CPO** | ChatGPT | Product, UX, AI, roadmap, strategy, priorities, user experience |
| **Lead Engineer** | Claude | Architecture, code quality, performance, infrastructure, scalability, security |

Lead Engineer has the right to veto if a decision destroys the architecture.  
After that, the decision returns to Product Review.

## Definition of Ready

No task begins development until:
- [ ] There is a product goal
- [ ] There is a UX design
- [ ] The user scenario is clear
- [ ] Approved by CPO

## Definition of Done

A feature is complete only if it:
- [ ] Works
- [ ] Is tested
- [ ] Is documented
- [ ] Does not degrade architecture
- [ ] Does not degrade UX
- [ ] Is ready for scaling

## Long-Term Goal

We don't want to become the biggest betting application.  
We want to become the most useful.

The user opens BetTracker AI not to record a bet.  
But because this is where they make their decision.

## The Most Important Rule

Every decision must answer one question:

> **"Will this make BetTracker AI objectively more useful to the user?"**

If no — we don't do it.
