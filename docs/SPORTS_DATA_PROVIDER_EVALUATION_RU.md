> **Тип:** Документ только для принятия решения (research-only). Без кода, без интеграции, без DB-схемы, без Scout v2.
> **Контекст:** `PRODUCT_VISION_GAP.md` (BetTracker AI → будущий LineHunter AI).
> **Цель:** Выбрать **стратегию провайдера спортивных данных для Phase 1** — гейт, который разблокирует calendar-driven Scout v2, глубокий анализ футбола/тенниса, market-aware вариации ставок, snapshot текущих коэффициентов, авто-сеттлмент, публичные превью и будущее обучение/калибровку.

## 1. Executive Summary (Краткое резюме)

**Рекомендуемая стратегия для Phase 1: взять семейство API-Sports как единственного основного провайдера — `API-Football` для футбола и `API-Tennis` для тенниса.**

Это единственный *доступный по цене* вариант, который покрывает **и футбол, и теннис** с **fixtures + результатами + pre-match/live коэффициентами + стабильными числовыми ID** у одного вендора, с одной моделью авторизации и единым форматом данных. Суммарная стоимость на бете — примерно **$40–$100/месяц** на оба спорта, против **$5 000–$10 000+/месяц** у enterprise-провайдеров коэффициентов/сеттлмента. [api-football pricing], [api-tennis pricing via SportsAPI], [lsports cost guide]

- **Лучший all-in-one вариант:** **API-Sports (API-Football + API-Tennis).** Одно семейство, оба спорта, fixtures/результаты/коэффициенты/ID, минимальная стоимость на бете. Это и есть рекомендация.
- **Лучшая split-стратегия, если перерастём:** **SportMonks (глубокий футбол, вкл. xG и premium-коэффициенты TXOdds) + API-Tennis (теннис)**, опционально усиленные **TheOddsAPI** для движения линий по нескольким букмекерам. Split повышает стоимость интеграции и вводит две системы ID, поэтому **откладываем** до момента, когда глубина футбольного анализа или гранулярность коэффициентов станут реальным узким местом. [sportmonks pricing], [sportmonks premium odds]
- **Enterprise-уровень (отложить):** **Sportradar** (официальные данные, 50+ спортов вкл. теннис+футбол) и **OpticOdds** (авто-грейдинг/сеттлмент ставок в реальном времени по 100+ букмекерам) — это эталоны качества/сеттлмента, но они за формой-запросом и стартуют от ~$5k–$10k+/месяц — возвращаемся к ним только когда точность авто-сеттлмента или «официальные коэффициенты» станут критичны для выручки. [sportradar pricing], [opticodds pricing]
- **Основные риски рекомендации:**
  1. **Задержка результатов** — финальные результаты в API-Football могут запаздывать **до 48ч** на соревнованиях с низким покрытием, что ограничивает авто-сеттлмент топ-лигами. [api-football coverage]
  2. **Глубина коэффициентов** — коэффициенты API-Sports подходят для snapshot, но по охвату букмекеров и движению линий беднее, чем SportMonks-Premium/TheOddsAPI/OpticOdds.
  3. **Два продукта, а не один эндпоинт** — футбол и теннис у API-Sports это отдельные продукты (отдельные ключи/квоты), хотя модель данных и соглашения общие.
- **Что нужно от founder/CPO:** утвердить API-Sports для Phase 1 и утвердить **ограничение объёма авто-сеттлмента** (только топ-лиги + безопасные рынки), описанное в §5 и §8.

## 2. Product Requirements (Требования к продукту)

Что данные Phase 1 обязаны поддерживать (выведено из `PRODUCT_VISION_GAP.md`):

| Требование | Почему важно | Минимальная планка |
|---|---|---|
| **Футбол: fixtures / календарь** | Scout v2 должен быть fixture-driven, а не LLM-фабрикация | Реальные предстоящие матчи по лиге + дате, на N дней вперёд |
| **Теннис: fixtures / календарь** | То же, для ATP/WTA/Grand Slam | Реальные предстоящие матчи с контекстом турнира/раунда |

| **Стабильные fixture ID** | Общий ключ Scout → Analyst → Bet → Settlement | Постоянный числовой/строковый ID, переиспользуемый между эндпоинтами |
| **Данные для глубокого анализа** | Football/Tennis Analyst v2 | Форма, H2H, таблицы, статистика; покрытие/рейтинг для тенниса |

| **Стоимость на бете** | До выручки | Низкая фиксированная месячная цена, без enterprise-минимума |
| **Путь к продакшн-масштабу** | После беты | Понятный тариф апгрейда или путь миграции |
| **Будущая совместимость** (mobile / public / i18n) | Публичные превью + мультиязычность | Данные должны кэшироваться и быть locale-нейтральными (ID/UTC) |

**Вне рамок этого документа:** выбор конкретных рынков для реализации, DB-схема, код синхронизации, логика Scout v2. Это идёт *после* решения по провайдеру.

## 4. Provider Deep Dives (Разбор по провайдерам)

### 4.1 API-Sports — API-Football (рекомендуемый основной, футбол)

- **Покрытие:** 1 200+ лиг/кубков; fixtures, livescores, таблицы, события, составы, игроки, статистика, прогнозы, pre-match + live коэффициенты; live-матчи обновляются каждые ~15с. [api-football coverage]
- **Стабильные ID:** постоянные числовые ID для fixture, лиги, команды, игрока — переиспользуемые во всех эндпоинтах (то самое свойство общего ID, нужное для Scout→Settlement).
- **Коэффициенты:** pre-match (`getOdds`, `getBookmakers`, `getBets`, `getMapping`) плюс in-play (`getodds/live`); поле `coverage` по каждому соревнованию сигнализирует, есть ли коэфф./статистика/составы для конкретного сезона лиги. [api-football odds/coverage]
- **Оговорка по результатам/сеттлменту:** соревнования без livescore держат статус `NS` и получают **финальный результат только через 48ч** после матча — то есть результаты не везде низколатентны. [api-football coverage]
- **Цены (опубликованы):** Free = 100 запросов/день; **Pro $19/мес = 7 500 запросов/день**; масштабируемые тарифы до ~1,5M запросов/день (custom). Rate-limit растёт с тарифом (Free 10 зап./мин → выше 300–1 200 зап./мин). [api-football pricing], [api-football ratelimit]
- **Вывод:** самый дешёвый вменяемый способ получить футбольные fixtures + результаты + коэффициенты + стабильные ID в одном месте. Идеальный футбольный «хребет» для Phase 1.

### 4.2 API-Sports — API-Tennis (рекомендуемый основной, теннис)

- **Покрытие:** ATP, WTA, ITF, Grand Slam / Masters / Challenger; livescore, расписание и результаты, статистика, таблицы, pre-match коэффициенты. [api-tennis review]
- **То же семейство, что и API-Football** — общие соглашения и стиль ID снижают стоимость интеграции против смешивания несвязанных вендоров.
- **Цены (по сообщениям):** Starter $40/мес (8 000 запросов/день), Premium $60/мес (80 000 запросов/день), Business $80/мес (200 000 запросов/день). [api-tennis pricing via SportsAPI]
- **Вывод:** прагматичный выбор для тенниса на бете — та же ментальная модель, что и футбол, низкая цена, покрывает планку fixtures/результаты/коэффициенты. Глубина коэффициентов беднее, чем у специализированных теннисных odds-фидов, что приемлемо для v1.

### 4.3 SportMonks (глубочайший футбол; **нет тенниса**)

- **Покрытие:** 2 200+ лиг, сильные live-данные и самый глубокий набор футбольной статистики (вкл. xG); эндпоинт States даёт чистый жизненный цикл матча для сеттлмента. [sportmonks football api]
- **Коэффициенты:** Standard odds-фид на всех тарифах; **Premium Odds-фид через TXOdds** — 42 рынка от 145+ букмекеров; in-play обновление ~1с, pre-match ~каждые 10 мин; история коэффициентов хранится до 7 дней после стартового времени, вкл. opening odds и историю изменений (полезно для движения линий). [sportmonks premium odds], [sportmonks odds faq]
- **Цены:** футбольные тарифы ~€39–€219/мес в зависимости от охвата лиг и региона (Европа vs Весь мир, Basic/Standard/Advanced). Годовые ~на 20% дешевле. [sportmonks pricing]
- **Блокер для нас:** **SportMonks сейчас не предлагает теннис** — то есть он в принципе не может быть решением с одним провайдером для наших двух спортов. Выбрать его = SportMonks (футбол) **+** отдельный теннисный провайдер = две системы ID. [sportmonks tennis coverage]
- **Вывод:** лучший *футбольный* апгрейд, если/когда глубина статистики или гранулярность коэффициентов API-Football начнут ограничивать Analyst v2. Откладываем, пока это не станет доказанным узким местом.

### 4.4 Sportradar (enterprise-эталон)

- **Покрытие:** 50+ спортов вкл. футбол и теннис, официальные лицензированные фиды, сравнение коэффициентов, низколатентные результаты, виджеты/редакционка — B2B. [sportradar coverage]
- **Цены:** публичных цен нет; enterprise-контракты обычно **$10 000+/месяц**, обсуждаются по спорту/объёму/масштабу; корректировки посезонно. [sportradar pricing], [lsports cost guide]
- **Вывод:** эталон качества и официальных данных, и правильный ответ на реальном масштабе или когда важны «официальные коэффициенты»/регуляторное доверие — но чрезмерен и дорог для поставленной на паузу пред-беты. Отложить.

### 4.5 TheOddsAPI (только агрегация коэффициентов)

- **Покрытие:** 11+ спортов вкл. футбол и теннис; ~40 мейнстрим-букмекеров (Bet365, DraftKings, FanDuel, William Hill…). Теннис — **в основном match-winner** с ограниченными spreads/totals. [theoddsapi tennis], [theoddsapi coverage]
- **Модель:** по кредитам — каждый вызов `/odds` стоит `рынки × регионы` кредитов, поэтому запрос многих рынков/регионов сжигает лимит быстро. Free = 500 кредитов/месяц; платные тарифы масштабируются (сторонний источник: ~$20 → ~$199/мес за 20k → 12M кредитов — **подтвердить на официальной странице цен**). [theoddsapi pricing (3rd-party)]
- **Нет глубины fixtures, нет результатов/грейдинга, нет статистики.** Это слой коэффициентов, а не «хребет».
- **Вывод:** опциональное **обогащение коэффициентами** позже — дешёвое движение линий по нескольким букмекерам для усиления market-aware bet builder — поверх «хребта» API-Sports. Не основной провайдер.

### 4.6 OpticOdds (коэффициенты в реальном времени + авто-сеттлмент)

- **Покрытие:** 100+ букмекеров по 25+ спортам вкл. футбол и теннис; pre-match, in-play, player props, травмы, составы, историч. коэфф., лимиты. [opticodds coverage]
- **Сеттлмент:** грейдит ставки в реальном времени по статусам **Won / Lost / Refunded / Half Won / Half Lost** в момент разрешения рынков — ровно тот движок авто-сеттлмента, который иначе пришлось бы строить самим. [opticodds grading]
- **Цены:** за формой-запросом, по сообщениям **от ~$5 000/месяц**. [opticodds pricing]
- **Вывод:** самый сильный ответ по *сеттлменту/market-aware*, и естественный апгрейд, когда надёжность авто-сеттлмента станет критичной для выручки. На бете откладываем из-за стоимости.

## 5. Auto-Settlement Implications (Последствия для авто-сеттлмента)

Авто-сеттлменту нужны три вещи от провайдера: **стабильный fixture_id**, **жизненный цикл result_status** и **надёжный финальный результат с timestamp**. При рекомендуемом «хребте» API-Sports:

- **Безопасные рынки v1 (авто-сеттлмент):**
  - Футбол **1X2 / Match Result** (П1 / X / П2) — детерминировано по финальному счёту.
  - Теннис **Match Winner** — детерминировано по результату завершённого матча.
- **Небезопасные / ручная проверка (v1):** азиатские форы, тоталы, обе забьют (BTTS), точный счёт, форы по сетам/геймам, player props, всё, что войдится при снятии/walkover/прерывании.
- **Требование к fixture_id:** *один и тот же* ID провайдера должен ставиться на объект в момент Scout и проноситься через Analyst → Bet → Settlement. Числовые ID API-Sports удовлетворяют этому внутри каждого спорта; **ID футбола и тенниса живут в разных пространствах имён**, поэтому храним `sport + provider_fixture_id` как составной ключ.
- **Жизненный цикл result_status:** маппинг статусов провайдера во внутренний enum — напр. `NS` (не начат) → `LIVE/1H/HT/2H` → `FT` (завершён) → `settled`; плюс аномальные терминалы `PST` (перенесён), `CANC`, `ABD` (прерван), `WO` (walkover, теннис), `RET` (снятие, теннис).
- **Пограничные случаи сеттлмента, обрабатываемые вручную в v1:** задержка результата ≤48ч на футбольных соревнованиях с низким покрытием (авто-сеттлмент только для лиг с покрытием livescore); снятия/walkover в теннисе (многие букмекеры войдят — **не** авто-сеттлить); перенесённые/переигранные матчи (результат никогда не приходит на исходную дату); корректировки счёта провайдером после сеттлмента (нужен audit trail + путь отката).
- **Правило confidence-score (v1):** авто-сеттлить только когда `market ∈ безопасный набор AND status == FT AND соревнование ∈ покрытие livescore`; всё остальное → очередь ручной проверки. [api-football coverage]

## 6. Market-Aware Bet Builder Implications (Последствия для market-aware конструктора ставок)

Может ли провайдер питать расчёт value/edge? Возможности по провайдеру для входов, нужных конструктору ставок:

| Вход по коэффициентам | API-Sports (Футбол/Теннис) | SportMonks (футбол) | TheOddsAPI | OpticOdds |
|---|---|---|---|---|
| Текущие pre-match коэфф. | ✅ | ✅ | ✅ | ✅ |
| In-play / live коэфф. | ✅ (футбол; ~15с) | ✅ (~1с) | ⚠️ ограниченно | ✅ |
| Атрибуция букмекера / источника | ✅ (по букм.) | ✅ (145+ через Premium) | ✅ (~40 букм.) | ✅ (100+ букм.) |
| Timestamp коэффициента | ⚠️ на основе snapshot | ✅ (история изменений) | ✅ | ✅ |
| Лучшая цена по букмекерам | ⚠️ выводить на клиенте | ✅ выводится из фида | ✅ | ✅ |
| Движение линий / opening→closing | ⚠️ ограниченно | ✅ (opening + история 7д) | ✅ | ✅ |
| Широта рынков (футбол/теннис) | ⚠️ основные рынки | ✅ 42 рынка (футбол) | ⚠️ теннис в осн. match-winner | ✅✅ глубоко |
| Качество входа fair-odds / edge | ⚠️ ОК для v1 | ✅ | ✅ | ✅✅ |

**Вывод:** API-Sports поддерживает **вменяемый v1 конструктора ставок** (текущие коэфф., по букмекерам, основные рынки, простая лучшая цена). Для сигналов **движение линий / closing odds / sharp-vs-public** — накладываем **TheOddsAPI** (дёшево) или переводим футбол на **SportMonks Premium** — но только когда продукт реально показывает эти сигналы пользователю. Не покупаем заранее глубину, которую Scout v2 ещё не использует.

## 7. Scout v2 Implications (Последствия для Scout v2)

Выбор провайдера — это то, что превращает Scout из *фабрикации* в *fixture-first*:

1. **Извлечь** реальные предстоящие матчи от провайдера (API-Football / API-Tennis), отфильтровав по спорту, лиге/туру, окну дат и важности.
2. **Прикрепить** коэффициенты + доступные рынки к каждому реальному матчу (сначала тот же провайдер; обогащение позже).
3. **Ранжировать** кандидатов по сигналам value/возможностей, рассчитанным по реальным коэффициентам (и, позже, по выходу модели).
4. **Объяснить** через AI **только после** получения реальных данных — LLM аннотирует и ранжирует; она никогда не выдумывает матчи или ID.
5. **Гарантировать**, что каждый кандидат Scout несёт реальный `sport + provider_fixture_id`, чтобы чисто перетекать в Analyst → Bet → Settlement.

Итог: работа LLM сужается с «генерировать матчи» до «объяснять и приоритизировать реальный список матчей от провайдера». Провайдер — источник истины; модель — слой комментария.

## 8. Recommended Phase 1 Architecture (Рекомендуемая архитектура Phase 1 — на уровне research, не реализация)

- **Выбор провайдера:** **API-Sports** — API-Football (футбол) + API-Tennis (теннис). Одно семейство, два ключа.
- **Коэффициенты:** начинаем со встроенных pre-match коэффициентов API-Sports. **Откладываем** обогащение TheOddsAPI и OpticOdds, пока конструктору ставок/сеттлменту реально не понадобятся более глубокие букмекеры или авто-грейдинг.
- **Каденс синхронизации (стартовая точка, подстраивать под rate-limit):**
  - Fixtures/календарь: тянуть скользящее **окно на N дней** несколько раз в день.
  - Коэффициенты: обновлять по расписанию перед стартом (плотнее ближе к старту); live-коэфф. только для матчей, активно находящихся в Scout.
  - Результаты: опрашивать завершённые матчи вскоре после ожидаемого конца, с **окном ретраев до 48ч** для футбола с низким покрытием. [api-football coverage]
- **Стратегия кэширования:** агрессивно кэшировать по ключу `sport + provider_fixture_id`; хранить **snapshot'ы opening/последних коэффициентов**, чтобы один и тот же матч/рынок переиспользовал прежний анализ (напрямую связано с гэпом кэширования стоимости AI в `PRODUCT_VISION_GAP.md`).
- **Хранить внутри:** fixtures, стабильные ID, статусы, финальные результаты, snapshot'ы коэффициентов и audit-записи сеттлмента. **Тянуть live:** in-play коэфф./счёт только при необходимости.
- **Первые рынки футбол/теннис:** футбол **1X2**; теннис **Match Winner** — безопасный набор для авто-сеттлмента.
- **Отложить:** азиатские форы, тоталы, props, точный счёт, форы по сетам/геймам; SportMonks/Sportradar/OpticOdds; функции live-трейдинга.

## 10. Final Recommendation (Финальная рекомендация)

- **Выбрать API-Sports (API-Football + API-Tennis) для Phase 1**, потому что это единственный недорогой вариант, дающий **fixtures + результаты + коэффициенты + стабильные ID и для футбола, и для тенниса** у одного вендора и в одной модели данных, за ~$40–$100/месяц — достаточно, чтобы разблокировать Scout v2, входы для глубокого анализа, v1 market-aware конструктора ставок и авто-сеттлмент безопасных рынков, без enterprise-контракта. [api-football pricing], [api-tennis pricing via SportsAPI]
- **Использовать TheOddsAPI для обогащения коэффициентами *только при необходимости*** — дешёвый способ добавить движение линий по нескольким букмекерам в конструктор ставок позже, поверх «хребта» API-Sports, не как основного провайдера. [theoddsapi pricing (3rd-party)]
- **Отложить SportMonks** до апгрейда глубины футбола (глубочайшая футбольная статистика + premium-коэффициенты TXOdds) — но помнить, что у него **нет тенниса**, поэтому это компонент split-стратегии, а не единое решение. [sportmonks premium odds], [sportmonks tennis coverage]
- **Отложить Sportradar и OpticOdds** до фазы масштаба/сеттлмента — лучшие в классе официальные данные и авто-грейдинг ставок соответственно, но ~$5k–$10k+/месяц и за формой-запросом; вернуться, когда точность авто-сеттлмента или официальные коэффициенты станут критичны для выручки. [sportradar pricing], [opticodds pricing], [opticodds grading]
- **Ограничить авто-сеттлмент в v1** топ-лигами + безопасными рынками (футбол 1X2, теннис Match Winner), потому что финальные результаты API-Football могут запаздывать до 48ч на соревнованиях с низким покрытием. [api-football coverage]

**Решение одной строкой:** *Взять API-Sports для Phase 1 (футбол + теннис); держать TheOddsAPI как опциональный слой коэффициентов, а SportMonks/Sportradar/OpticOdds — как названные пути апгрейда для глубины, официальных данных и авто-сеттлмента.*

> **Пока не превращать это в план реализации.** Сначала нужно решение по провайдеру. Build plan идёт после утверждения §10.
